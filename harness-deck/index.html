The Big Idea

Tab 1 — Morning Brief

  • Pulls every signal source in parallel at 7AM
  • Scores each item against your profile, drops the noise
  • Reads your next 48h of calendar to shape the brief
  • Ends with an action queue: who to email today

Tab 2 — Outbound Engine (MedMorphIQ)

  • Receives the action queue from Tab 1
  • Enriches contacts, drafts emails in your voice
  • Everything queues for review — nothing auto-sends
  • Suggests meeting slots from your real calendar

How a Morning Run Works

Each stage checkpoints to SQLite before moving on. If something breaks mid-run, it replays from the last checkpoint — not from scratch.

7:00 AM
Fetch
All sources in parallel. Alarm if < 3/8 return data.
STAGE 2
Score
Claude scores each item 0.0–1.0. Drop below 0.4.
STAGE 3
Calendar
Read 48h. Surface meeting prep. Adjust length.
STAGE 4
Generate
Claude writes the brief. Every name cites a source.
STAGE 5
Actions
Who to email + why. Matching jobs. Overdue follow-ups.
STAGE 6
Deliver
Render to UI. Persist run. Emit OTel spans.
05

The Four Pillars

The whole system stands on these four. Every design decision traces back to one of them.

Pillar 1

The Agent Loop

The harness drives the agent. The agent never calls the model directly. Hard limits the agent cannot override:

  • MAX_TURNS = 8 — can't think forever
  • TOKEN_BUDGET = 32k — can't blow the context
  • WALL_CLOCK = 120s — can't hang the morning
def run(user_input, max_turns=8):
    messages = [{"role": "user", ...}]
    for _ in range(max_turns):
        reply = client.chat(messages, tools=TOOLS)
        if not reply.tool_calls:   # done
            return reply.text
        for call in reply.tool_calls:
            dispatch(call)  # guardrails here
    raise RuntimeError("turn limit")  # alarm
Pillar 2

Tools

Every data source and action is a typed tool. The agent reads the schema, decides when to call it, and always gets a predictable response — errors are data, not crashes.

  • Schema — what it does
  • Executor — the real code
  • Result Contract — always {ok, data, error}

Examples

  • search_news — capped at 2000 tokens
  • read_gmail — read-only
  • enrich_contact — 10 lookups/run max
  • send_email — hard block without approval token
Pillar 3

Guardrails

Three layers. Declared as named constants in guardrails.py — not baked into prompts.

  • Input — drop <0.4 relevance, dedup 48h, cap 24k tokens, allow-listed domains
  • Action — tool must be allow-listed, send_email needs token, max 10 enrichments
  • Output — every named entity needs a citation, schema validated, low-confidence drafts flagged
Pillar 4

Observability

Every model call and tool dispatch emits a structured trace span. Four signals per run, streamed to Langfuse (free tier) — replay any run, score output quality.

  • p95 latency
  • $ / run cost
  • err % rate
  • eval pass rate

Structured alarms carry type · severity · context · recommended action — see the table below.

08

Human-in-the-Loop

Three places the harness stops and asks instead of guessing.

1 · Review Gate
All outbound drafts queue here. Approve / Edit / Reject before anything sends.
2 · Yellow Flag
Brief items scored 0.40–0.55. "We think this is relevant — verify."
3 · Alarm Triage
HIGH + CRITICAL alarms pause the run. Needs acknowledgment before it continues.
09

Calendar Intelligence

The calendar read happens before brief generation and shapes the entire output.

Meeting prep

  • Call with someone today? Their recent news and context surface at the top of your brief.

Schedule-aware length

  • Packed day → shorter brief, more aggressive filtering.

Action-to-calendar

  • Follow-up tasks get suggested as calendar blocks based on your real availability.

Outbound timing

  • When the outbound agent drafts an email, it can suggest a slot pulled from your actual calendar.
